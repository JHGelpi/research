## TL;DR


At its core, a data platform is the foundation that allows a team to collect, transform, store, and access information from across the business.
 Building one for internal users requires deliberate decisions across five functional layers — ingestion, storage, transformation, orchestration, and serving — layered with governance, observability, and an appropriate team operating model. 
In 2024–2025, the landscape is increasingly shaped by decentralized (data mesh) architectures, FinOps-conscious cost management, and the integration of GenAI/LLM use cases.

## Key Findings

### 1. The Five Core Layers of a Data Platform


A data platform, also referred to as a "data stack," is composed of five foundational layers: data storage and processing, data ingestion, data transformation, business intelligence (BI) and analytics, and data observability.


**Layer 1 — Ingestion**

The ingestion layer moves data from sources to downstream systems, either in real time or in batches. Source systems include databases, SaaS applications, APIs, log files, or IoT devices.
 
Tools like Apache Kafka, Fivetran, and AWS Kinesis operate here, efficiently collecting, streaming, or forwarding data from a wide range of sources.
 
Current best practice is to leave data in its raw format during ingestion so it can be analyzed in different ways without having to re-ingest from the source.



Batch processing is the most common form of ingestion — it collects and groups data into batches sent to storage, and is typically used when real-time data is not necessary because it requires less work and is less expensive.
 
Real-time (stream) processing, by contrast, obtains, transforms, and loads data as it is recognized.


**Layer 2 — Storage**

A data warehouse primarily handles structured data and excels at business intelligence and reporting, while a data lakehouse combines features of both data warehouses and data lakes, supporting all data types and enabling both traditional analytics and ML.


- 
A data lake stores raw, unstructured data at scale, ideal for machine learning and real-time ingestion; a data warehouse stores cleaned, structured data optimized for fast querying and BI.

- 
A data lakehouse combines the flexibility and low-cost scalability of a data lake with the performance, structure, and governance of a data warehouse; it supports open table formats like Delta Lake, Apache Iceberg, and Apache Hudi.

- 
However, lakehouses also introduce complexity: integrating components across the stack, tuning performance, and achieving cost predictability remain challenges for many organizations.


**Layer 3 — Transformation**

Data transformation facilitates the cleaning, processing, and enrichment of data, preparing it for consumption by analytical tools and applications.
 
The transformation layer deals with changing the structure and format of data — for example, converting unstructured data to SQL format to make it easier to search.



dbt excels at SQL-based transformation, AI-assisted development, and enterprise governance through state-aware execution and semantic layers.
 
dbt empowers anyone with SQL knowledge to own the logic behind data transformations, giving them control over cleansing data and delivering actionable insights.


**Layer 4 — Orchestration**

Data orchestration has become crucial and not only streamlines the platform-building process but also offers scalable options to align with the trajectory of any business; orchestration is commonly executed through Directed Acyclic Graphs (DAGs) that structure hierarchies, dependencies, and pipelines of tasks across multiple systems.



Apache Airflow is an open-source platform that allows teams to programmatically author, schedule, and monitor workflows; using DAGs, Airflow lets you define dependencies between tasks, manage retries, and track progress — and is designed for scalable, repeatable, and auditable workflows.


A common pattern is to combine both tools: 
use dbt to implement transformations, tests, and documentation inside the warehouse with a model-centric approach that yields clean, version-controlled datasets; use Airflow to orchestrate the broader flow — triggering ingestion, kicking off dbt jobs, running ML training, and managing downstream exports.


**Layer 5 — Serving**

The data serving layer ensures that processed data is made available to users and applications in an efficient and accessible manner, often through APIs, dashboards, or other interfaces.
 
The serving layer often includes a cloud data warehouse where relational tables are stored, but may also include data lake, data lakehouse, or data mart deployments.

### 2. Storage Architecture Decision


Data warehouses offer fast, structured analytics; data lakes provide flexible, low-cost storage; and lakehouses combine the best of both.
 
AI, real-time analytics, and open architectures are driving the need for scalable, governed, and interoperable platforms — and unified lakehouse platforms merge the scale of lakes with the performance of warehouses, reducing complexity and supporting diverse use cases.


---

### 3. Data Governance & Access Control


Data governance platforms are specialized software solutions designed to establish, enforce, and maintain data policies, standards, and practices throughout an organization; they centralize processes for data quality, security, usability, compliance, and accountability — and typically integrate tools for data cataloging, lineage tracking, access control, and workflow automation.



A robust governance strategy involves implementing role-based access control (RBAC), data masking for sensitive fields, and encryption for data both in-motion and at-rest.



Identity and Access Management (IAM) platforms control who can access which systems, while more granular access control tools manage permissions at the data level (e.g., which users or roles can see specific tables, columns, or rows) — these tools are critical for enforcing the principle of least privilege.



Metadata is "data about your data" — it describes the origin, format, and business context of data assets; lineage provides a complete audit trail showing where data came from, how it was transformed, and where it is going; together, they are essential for impact analysis and building trust in data.



The best data governance tools in 2025 combine metadata management, access governance, and compliance automation — monitoring who can access which datasets, tracking how permissions evolve, and detecting overexposed or orphaned identities.

### 4. Data Observability


Gartner defines data observability tools as software that enables organizations to "understand the state and health of their data, data pipelines, data landscapes, data infrastructures, and even the financial cost of data." These platforms achieve this by continuously monitoring, tracking, alerting on, analyzing, and troubleshooting data issues.



These platforms offer extensive monitoring across five essential pillars: data quality, lineage tracking, freshness monitoring, volume tracking, and schema evolution oversight.



Monte Carlo is an industry-leading data observability platform that monitors freshness, volume, schema, and quality incidents across warehouses and BI tools with ML-powered anomaly detection.
 Other notable tools include 
Soda (combining an open-source testing framework with a SaaS control plane) and Datafold, which specializes in data diffs for CI/CD workflows.



Open-source projects like Soda Core, Elementary Data, and dbt Tests are democratizing capabilities once locked behind expensive platforms, and the convergence of data quality testing and data observability is creating a new expectation: reliable, transparent data pipelines built the same way modern software systems ensure reliability.

### 5. Organizational Models & Team Structure


What was once a small group of analysts generating reports has expanded into cross-functional teams of engineers, scientists, and product managers who own everything from ingestion to BI; data teams are evolving with lean core teams expanding into hybrid models that mix centralized engineering with embedded analysts, and roles are diversifying with analytics engineers and data product managers rising.


**Centralized vs. Data Mesh:**

Organizations are shifting towards a data mesh architecture, decentralizing data ownership and control — this approach distributes data management responsibilities across various domains, enabling more agile and responsive data handling.



In a data mesh model, a platform team builds and maintains the self-serve data infrastructure that all domain teams use; they are responsible for the tooling, templates, environments, and APIs that make it possible for domain teams to build and publish data products.



In a data mesh framework, teams own not only their own data but also the data pipelines and processes associated with transforming it; a central data engineering team maintains both key datasets and a suite of self-service tools to enable individual data ownership, and domain-specific data teams then exchange data via well-defined and versioned contracts.


**Team Sizing:**

Four to eight engineers can run a serious platform serving 100–300 analytics and AI engineers if the platform is well-designed; platforms requiring 20+ platform engineers usually have accumulated technical debt that limits leverage.


**Self-Serve Analytics:**

Self-serve analytics in 2025 is about empowering everyone in your organization to access and analyze data without requiring technical skills.
 
Governance through automation enables self-service — the framing of "trade off governance for self-service" usually means governance has been implemented as approval gates rather than as automated controls.

### 6. Emerging Trends

- **AI Integration:** 
AI is operating autonomously in cloud-native data warehouses, automatically identifying patterns, anomalies, and insights with less human oversight, allowing data teams to focus on strategic initiatives rather than routine data analysis.

- **Open Table Formats:** 
A traditional data warehouse suffered from being an OLAP data store situated separately from operational systems, forcing data segregation and creating data silos; open table formats like Apache Iceberg are addressing this.

- **Platform Consolidation:** 
Modern providers of data warehouse technology typically offer a suite of other tools such as ELT connectors, dashboarding services, managed Spark, basic orchestration, in-built data quality monitoring, alerting, and tools focused around ML and AI.

- **AI/Analytics Team Friction:** 
A documented failure pattern is when an AI team bypasses the official platform because onboarding a new data source takes 4–6 weeks; they needed 4–6 hours — the platform was built for analytics workloads, but AI workloads needed a different shape of self-service.
